# ShikkeeOS System Map

This document defines the first working map of ShikkeeOS.

The goal is simple:

Turn a fragmented housing journey into one integrated system.

---

## System overview

ShikkeeOS connects five core layers:

1. Customer layer
2. Configuration layer
3. Pricing layer
4. Infrastructure layer
5. Deployment layer

These layers should behave as one product, not as separate tools.

---

## 1. Customer layer

Purpose:
The buyer-facing layer where users discover, configure, and move forward.

Main responsibilities:
- show home models
- show options and add-ons
- guide customer flow
- collect intent and purchase inputs
- reduce confusion at every step

Outputs:
- selected model
- selected options
- customer profile
- ownership path
- purchase intent state

---

## 2. Configuration layer

Purpose:
Turn customer choices into a structured home configuration.

Main responsibilities:
- map available models
- validate combinations
- attach add-ons
- define allowed configurations
- keep choices system-safe

Outputs:
- configuration object
- add-on selections
- build constraints
- final product package

---

## 3. Pricing layer

Purpose:
Translate configuration into fixed-price logic.

Main responsibilities:
- price model selection
- price options and add-ons
- preserve pricing rules
- reduce quote ambiguity
- keep pricing understandable

Outputs:
- base price
- option pricing
- bundled pricing
- total system price

---

## 4. Infrastructure layer

Purpose:
Integrate energy and infrastructure logic into the product itself.

Main responsibilities:
- solar-ready planning
- battery-ready logic
- infrastructure compatibility
- system-native add-on rules
- energy optimization inputs

Outputs:
- infrastructure package
- energy configuration state
- add-on compatibility result

---

## 5. Deployment layer

Purpose:
Connect the sold configuration to delivery and on-site assembly.

Main responsibilities:
- translate order into delivery workflow
- define assembly-ready package
- support timeline clarity
- reduce contractor fragmentation
- maintain system continuity from click to delivery

Outputs:
- deployment package
- timeline state
- delivery readiness state

---

## Operating principle

The customer should not experience five systems.

They should experience one product.

That is the point of ShikkeeOS.

---

## Core design rule

Every layer must reduce chaos.

If a layer adds confusion, delay, or ambiguity, it is failing.
